# ptrade Wyckoff 风险登记

## 目的

这份文档记录 ptrade 路线当前最关键的不确定性。

它不是路线宣传，也不是功能清单。它用于回答三个问题：

1. 当前功能方向是否偏离 Wyckoff 交易理念。
2. 当前实现距离 ptrade 实盘运行目标还差什么。
3. 数据层缺口是否足以阻断模型继续推进。

## 总体判断

当前方向没有根本偏离 Wyckoff。策略仍围绕 Phase A/B/C/D、Spring/ST/SOS/LPS/BUEC/UTAD、支撑阻力、量价、RS/Beta、L2/CVD、试仓到正式仓推进。

但当前还不能称为“可实盘的 Wyckoff 模型”。更准确的定位是：

> 一个运行在 ptrade 内的 Wyckoff 研究与模拟盘验证框架，正在从启发式信号系统推进到可复核的结构候选系统。

当前最大风险不是写不出交易脚本，而是：

- Wyckoff 结构判断是否能被稳定编码成可复核证据对象。
- A 股 ptrade 环境能否提供足够稳定、足够新鲜的 L2 / 逐笔数据。
- 模拟盘订单、成交、持仓、报告和状态记忆能否闭环一致。
- 是否能用足够多的历史样本证明系统不是在放大启发式误判。

## 可行性边界

### 可以继续做

- 继续做回测、模拟盘、报告和状态记忆闭环。
- 继续把策略从最终信号改造成 `candidate + evidence + rejectionReasons + invalidation`。
- 继续把 L2 / 逐笔作为 Spring 真伪、UTAD 假突破和 LPS 承接质量的确认层。
- 继续用人工复核作为研究校准，不作为 live trade 审批层。

### 暂时不能承诺

- 不能承诺当前启发式已经足以稳定识别 Wyckoff Phase C / D。
- 不能承诺无 L2 / 逐笔时仍能可靠区分 Spring、反抽失败和 UTAD。
- 不能承诺当前 ptrade 账号在真实交易时段一定有可用 L2 / 逐笔权限。
- 不能承诺当前报告已经能解释所有订单、成交、持仓和账户级偏差。
- 不能承诺当前版本能进入自动实盘。

### 应该暂停或降级的条件

如果出现下面任一情况，应暂停 live armed 方向，退回研究 / 模拟盘：

- 模拟盘报告与实际成交、持仓或账户级订单持续对不上。
- 真实交易时段无法稳定拿到 L2 / 逐笔，且无法区分无权限、空数据、延迟或字段不兼容。
- 20 个以上 A 股 Phase C / D 复核窗口里，系统不能稳定区分“真实收回后的 LPS / SOS 延续”和“破位反抽 / UTAD / 证据不足”。
- 撤单、部分成交、重启恢复或次日对账任一环节无法形成可审计闭环。
- 结构候选仍只能给最终买卖信号，不能输出证据、拒绝原因和失效条件。

## 风险清单

### R1：结构语义偏移

- 等级：高
- 风险：当前 `Phase E` 被用于失败结构 / SOW 跌破后的状态。这容易和 Wyckoff 积累完成后的上涨 Phase E 混淆。
- 影响：复盘语言会失真，后续样本标注和分类统计会被污染。
- 当前证据：canonical 策略中 `WEAKNESS` 和“跌破后待自动反弹”仍使用 `Phase E`。
- 应对：把失败路径独立为 `Failure/SOW` 或 `DistributionRisk`，不要复用 Phase E。

### R2：Wyckoff 候选对象不足

- 等级：高
- 风险：当前输出仍偏最终 `status` 和仓位决策，候选证据对象不完整。
- 影响：错误发生时无法判断是结构识别错、数据确认错，还是执行恢复错。
- 缺口：缺 `candidate`、`evidence`、`rejectionReasons`、`invalidation`、`reviewLabel`、`causeCount`。
- 应对：先把策略输出改成结构候选，再决定 `PILOT_ACTION` / `ACTION_REQUIRED`。

### R3：cause count 仍是近似值

- 等级：高
- 风险：当前 `pnfColumnCount` 本质是 Phase B/C 区间持续计数，不是真正 P&F cause count。
- 影响：目标价、风险收益比和持仓推进可能被伪精度误导。
- 应对：短期在报告中标记为 `causeProxy`；中期引入明确的 P&F / 横向栏数算法或人工标注对照。

### R4：模拟盘闭环未现场验证

- 等级：高
- 风险：主推、轮询、账户级订单、成交、持仓、报告和状态记忆还没有在真实模拟盘中证明一致。
- 影响：不能进入 live 评估。
- 应对：用同一份 canonical 脚本跑一轮模拟盘，逐项核对 `orderResponseEvents`、`tradeResponseEvents`、`orders`、`trades`、`positions`、`accountAudit` 和 pickle state。

### R5：订单状态机不完整

- 等级：高
- 风险：当前主推回调只记录事件，不驱动完整订单状态机。
- 影响：部分成交、拒单、废单、撤单确认和重复挂单恢复仍可能解释不清。
- 应对：建立 `submitted -> accepted -> partial -> filled -> cancel_requested -> canceled/rejected/unknown` 状态机，并持久化订单身份。

### R6：撤单恢复还缺重报价和失败升级

- 等级：高
- 风险：当前已有 `cancel_order` 超时撤单，但没有撤单确认、重报价策略、冷却期、失败升级。
- 影响：模拟盘和实盘中可能卡在 stale open order，或撤单后重复追价。
- 应对：先只对本策略订单做恢复；非本策略订单只报告不自动撤。

### R7：L2 / 逐笔权限和数据新鲜度未知

- 等级：高
- 风险：真实交易时段是否能稳定拿到 L2 盘口和逐笔成交仍未验证。
- 影响：Spring 真伪、UTAD 假突破、LPS 承接质量无法可靠确认。
- 应对：真实交易时段记录字段样例、时间戳、延迟、空数据原因、权限错误和订阅边界。

### R8：微观确认默认可降级

- 等级：中
- 风险：`require_l2_for_entry=False`、`require_trade_stream_for_entry=False` 会允许缺少微观确认时进入 paper/sim 候选。
- 影响：Phase 1 可跑通，但 Phase C / D 质量会被削弱。
- 应对：报告必须区分 `dataAvailable`、`confirmed`、`gateReady`；只有真实权限验证通过后再切 strict gate。

### R9：A 股样本校准不足

- 等级：高
- 风险：当前没有足够多的 A 股 Phase C / D 标注窗口来验证模型。
- 影响：回测可能只是参数碰巧适配，不能证明模型有效。
- 应对：建立至少 20 个历史窗口，覆盖 Spring 成功、Spring 失败、UTAD、破位反抽、LPS/SOS 延续和证据不足。

### R10：RS / Beta 与政策池仍不够可审计

- 等级：中
- 风险：RS/Beta 默认不强制，政策池是静态 JSON。
- 影响：与“长周期背景与相对强度没有失真”的目标仍有距离。
- 应对：记录数据来源、更新时间、入池理由、剔除理由；决定何时将 RS/Beta 从加分项升为阻断项。

### R11：A 股交易约束不完整

- 等级：中
- 风险：已有 100 股和可卖数量保护，但涨跌停、停牌、ST、退市风险、价格档位、集合竞价、手续费、滑点和 T+1 次日确认仍不完整。
- 影响：模拟盘和实盘成交行为可能与回测目标数量不一致。
- 应对：把这些约束放入执行前检查和日终对账，而不是只体现在仓位计算里。

### R12：重启恢复不足

- 等级：中
- 风险：部分订单身份只存在内存里，重启后可能丢失。
- 影响：账户级订单可能被误标为 unmatched，恢复逻辑可能失去上下文。
- 应对：把本策略订单 id、entrust no、最后主推事件、最后巡检摘要写入状态记忆或独立执行日志。

## 下一步风险收敛顺序

1. 回测结构审查：先证明历史回测输出的候选没有偏离 Wyckoff 结构语义。
2. 结构候选对象改造：证明 Wyckoff 判断可复核。
3. A 股样本集复核：证明模型不是启发式误判集合。
4. 模拟盘闭环验证：证明执行链路可信。
5. L2 / 逐笔真实交易时段验证：证明微观数据可信。
6. 订单状态机和持久化：证明恢复链路可信。
7. 次日对账与风控闸门：证明 live 评估有基础。

## 当下的真实判断

现在怀疑能不能做出来是合理的。

如果目标是“自动读懂 Wyckoff 并稳定实盘赚钱”，当前证据不够，不能承诺。

如果目标是“把 Wyckoff 交易过程拆成可观测证据、候选分类、模拟盘执行和风险闸门，再逐步验证哪些环节有效”，当前路线仍然可推进。

下一阶段不应该继续堆信号，而应该用模拟盘和样本复核来回答一个硬问题：

> 这套系统是否真的能稳定区分可右侧验证的结构修复，和看起来像机会但实际不可交易的反抽 / 假突破 / 证据不足。
