# ptrade Wyckoff 目标与成功准则

## 最终目标

建立一套运行在 ptrade 内部的 A 股 Wyckoff 结构交易系统：

- 能用长周期背景、结构阶段、RS / Beta、L2 盘口和逐笔成交去区分“可右侧验证的结构机会”与“不可交易的反抽 / 假突破 / 失败结构”。
- 能在 ptrade 的实际约束下完成回测、模拟盘、执行恢复、日终报告和状态记忆闭环。
- 先支持可复核的回测与模拟盘 paper trade，最后才评估带审批与风控闸门的受控自动化执行。

最终系统必须能把下面这句话变成可复核的数据对象：

> 当个股完成 Phase C / D 的结构修复、长周期背景与相对强度没有失真、微观确认没有恶化，并且在 A 股最小成交单位、T+1 和订单状态约束下仍存在可执行的右侧入场点时，系统才允许进入模拟盘或受控执行候选。

## 非目标

- 不做左侧抄底脚本。
- 不用单根最低点或理想化 K 线价格假设成交。
- 不把“无 L2 / 逐笔时可降级运行”等同于“微观确认已完成”。
- 不在 `cancel_order` 恢复、次日对账、审批和风控闸门完成前进入自动执行。
- 不把 ptrade 当前版本描述成无人工监管的实盘交易系统。

## 成功准则

### SC-1 运行时能力基线

完成标准：

- 策略能稳定解析运行时股票池、标的代码和回测 / 模拟盘上下文。
- `order`、`get_open_orders`、`get_orders`、`get_trades`、`get_positions`、状态记忆和日终报告链路全部可验证。
- 报告能明确区分 active symbols、运行来源、执行模式和能力降级状态。
- flat-start、标的池切换和已有状态文件残留时，state memory 会自动收敛。

失败条件：

- 当前运行标的来源不明确。
- 同一份策略在回测和模拟盘之间出现上下文歧义。
- 报告无法解释状态为何延续、为何回收或为何降级。

### SC-2 执行闭环真实性

完成标准：

- 模拟盘能够完成订单、成交、持仓、报告和状态记忆闭环。
- 模拟盘闭环不仅依赖轮询查询，还能解释 `on_order_response` / `on_trade_response` 与 `get_orders` / `get_trades` / `get_positions` / `get_all_orders` 之间的状态差异。
- 报告能对齐实际 open orders、filled trades、positions 和 strategy state。
- 下单逻辑显式考虑 A 股最小 100 股手数、部分成交、重复挂单防护和 T+1 约束。
- 超时未成交场景有 `cancel_order` 撤单 / 重报价和失败恢复策略。

失败条件：

- 模拟盘报告与实际成交 / 持仓对不上。
- 仍用理想目标仓位替代可执行数量。
- 异常挂单或部分成交后没有恢复路径。

### SC-3 微观确认与数据契约

完成标准：

- 明确 ptrade 环境中的 L2、逐笔成交、权限和订阅边界。
- 明确当前券商的 Python 版本、关键 API 返回格式和可用三方库边界，至少覆盖 `python3.5 / python3.11` 差异与 DataFrame / Panel / dict 返回差异；当前已知券商为国金，默认优先按 `python3.11` 路径验证。
- L2、逐笔、订单、成交、持仓和状态记忆都有统一字段定义，而不是直接耦合原始返回。
- 报告能区分 `dataAvailable`、`confirmed`、`gateReady`、`capabilityUnavailable` 和临时空数据。
- 外部因子、HTTP 和第三方库能力必须被视为可选增强，而不是默认可用能力；默认按 ptrade 内网、不可自由 `pip install` 约束设计。
- 即便当前券商为国金，也不能直接把“国金通常是 `python3.11`”当成最终事实，仍需区分测试版 / 实盘版客户端和账户权限差异。
- 在真实交易时段完成一轮权限验证后，才能决定是否切换 strict micro gate。

失败条件：

- UI 或策略逻辑直接依赖未标准化的原始 payload 字段。
- 无法判断 L2 / 逐笔缺失是权限问题、交易时段问题还是接口能力问题。
- 在未验证真实权限前就把微观确认切成强制闸门。

### SC-4 结构候选过滤器

完成标准：

- 策略先输出结构候选与拒绝原因，再决定是否允许 `PILOT_ACTION` 或 `ACTION_REQUIRED`。
- 候选对象至少包含阶段、支撑 / 阻力、长周期背景、RS / Beta、L2 / CVD 结论、风险拦截原因和失效条件。
- 系统能区分“真实收回后的 LPS / SOS 延续”“反抽失败”“UTAD / weakness 风险”“证据不足”。
- cause count、次级测试和失败结构识别进入报告对象，而不是只体现在最终仓位结果里。

失败条件：

- 只因价格反弹或突破就直接给正式执行信号。
- 报告不能解释为什么一个候选被拦截。
- 结构判断和执行判断混在一起，无法复盘错误来源。

### SC-5 Paper Trade 与执行闸门

完成标准：

- paper trade 只在结构和微观证据通过后进入。
- 每笔候选都记录入场理由、止损、目标、订单簿环境、数据新鲜度、审批状态和执行结果。
- 完成 `get_deliver()` / `get_fundjour()` 次日对账后，才能评估更进一步的自动化执行。
- live execution 必须晚于模拟盘闭环、撤单恢复、对账和审计日志。

失败条件：

- 没有审批或风控闸门就进入 live armed。
- 数据延迟、无心跳、无对账时仍继续生成执行动作。
- 无法区分策略判断错误和执行恢复错误。

## ptrade 硬闸门

- 无模拟盘订单 / 成交 / 持仓 / 报告闭环，不进入 live 评估。
- 无真实交易时段 L2 / 逐笔权限结论，不切 strict micro gate。
- 无当前券商 Python 版本、关键 API 返回格式和运行约束确认，不扩大 tick / 线程 / 多返回类型耦合实现。
- 无 `cancel_order` 超时撤单 / 重报价，不允许进入自动执行。
- 无 `get_deliver()` / `get_fundjour()` 次日对账，不允许进入自动执行。
- 无审批、风控和审计日志，不允许从 paper trade 进入 live armed。

## 第一阶段北极星指标

第一阶段只看一个核心指标：

> 对 A 股历史 Phase C / D 样本，系统能否稳定区分“真实收回后可右侧验证的 LPS / SOS 延续”与“破位反抽、UTAD 或证据不足的失败结构”，并且只在存在可执行数量时进入模拟交易候选。

如果这个指标不成立，后续的 runner 管理、P&F 目标、仓位优化和自动执行都只是在放大误判。
