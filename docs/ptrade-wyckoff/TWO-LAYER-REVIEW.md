# ptrade 双层审查方案

## 目的

ptrade 路线后续按两层审查推进：

1. 回测结构审查：先确认策略能不能在历史数据里识别出可复核的 Wyckoff 结构。
2. 模拟盘执行审查：再确认同一套策略能不能在 ptrade 模拟盘里跑通订单、成交、持仓、报告和状态记忆闭环。

这两层不能混在一起判断。否则模拟盘结果不好时，无法判断问题来自结构识别、数据缺口，还是执行链路。

## 第一层：回测结构审查

核心问题：

> 回测阶段输出的候选，是否真的像 Wyckoff 结构，而不是追涨杀跌或参数拟合。

审查对象：

- Phase A / B / C / D 识别是否合理。
- Spring / Test / SOS / LPS / BUEC / UTAD 是否有清晰证据链。
- 失败结构是否能被拒绝或降级，而不是硬归类成机会。
- 长周期背景、交易区间、支撑阻力、成交量行为、RS / Beta 是否被纳入判断。
- 缺少 L2 / 逐笔时，报告是否明确标记微观确认不可用，而不是把缺口当成通过。

最低输出要求：

- `candidateType`：候选类型，例如 `spring`, `sos`, `lps`, `utad`, `breakdown_retest`, `insufficient_evidence`。
- `phaseContext`：Phase A / B / C / D 或失败结构背景。
- `evidence`：支持候选的量价、区间、长周期、相对强弱证据。
- `rejectionReasons`：否决或降级原因。
- `invalidationLevel`：结构失效价格或条件。
- `confidence`：只作为排序和复核线索，不作为自动实盘依据。
- `dataGaps`：缺少 L2、逐笔、RS/Beta 或其他关键数据时必须显式记录。

验收方式：

- 建立至少 20 个 A 股历史结构窗口。
- 样本需覆盖 Spring 成功、Spring 失败、SOS/LPS 延续、UTAD、破位反抽、证据不足。
- 每个窗口保留算法输出、人工复核结论和差异说明。
- 第一轮不以收益为主指标，先以结构识别是否可解释、可复核为主。

通过标准：

- 多数候选能给出清楚证据链和失效条件。
- 失败样本可以被拒绝、降级或解释为证据不足。
- 人工看图和算法候选在主要结构方向上大体一致。
- 报告能把“结构判断”和“数据缺口”分开呈现。

暂停或降级标准：

- 候选只有最终买卖信号，没有证据链。
- Phase C / D、Spring、UTAD、破位反抽经常混淆。
- 20 个历史窗口中大量样本只能靠事后解释才能成立。
- 缺少 L2 / 逐笔时仍把微观确认写成通过。

## 第二层：模拟盘执行审查

核心问题：

> 当结构层给出候选后，ptrade 模拟盘能不能稳定执行、记录、对账和恢复。

审查对象：

- `on_order_response` / `on_trade_response` 主推回调是否完整记录。
- `get_open_orders` / `get_orders` / `get_trades` / `get_positions` 轮询结果是否与主推一致。
- `get_all_orders` / `get_all_positions` 是否能识别账户级偏差、人工单和非策略持仓。
- 下单、部分成交、拒单、撤单、持仓变化、状态记忆是否能形成一条可审计链路。
- 重启后是否能通过状态文件和账户巡检恢复上下文。
- 报告是否能解释每笔订单从候选信号到执行结果的全过程。

最低输出要求：

- `orderResponseEvents`
- `tradeResponseEvents`
- `orders`
- `openOrders`
- `trades`
- `positions`
- `accountAudit`
- `executionReconciliation`
- `strategyState`

验收方式：

- 先跑 1 个完整模拟盘交易日，按 `PAPER-TRADE-ACCEPTANCE.md` 验收。
- 通过后再连续观察 3 到 5 个交易日。
- 使用 `npm run ptrade:paper-report:check -- --report=<report.json>` 做报告格式和一致性检查。
- 对部分成交、无成交、回调缺失、账户持仓不一致等场景补本地 fixture。

通过标准：

- 报告中的主推、轮询、账户级巡检可以互相解释。
- 策略内订单和账户级订单能区分清楚。
- 持仓变化与成交记录一致。
- 状态记忆重启后不丢失关键订单身份。
- 异常情况能进入拒绝、降级、暂停或人工复核状态，而不是继续自动推进。

暂停或降级标准：

- 模拟盘报告持续对不上真实成交或持仓。
- 订单状态无法解释部分成交、撤单或拒单。
- 重启后无法恢复本策略订单身份。
- 账户级偏差只能发现，不能形成明确处置或隔离策略。

## 推进顺序

1. 先做回测结构审查，确认模型方向没有偏离 Wyckoff。
2. 同步准备模拟盘报告验收和本地执行 fixture，但不把模拟盘盈亏当成第一判断。
3. 回测结构审查通过后，再用模拟盘审查 ptrade 执行闭环。
4. L2 / 逐笔数据作为第三层增强确认，在真实交易时段验证权限和新鲜度后再决定是否强制。

## 当前结论

当前下一步不是直接证明策略能赚钱，而是先回答两个更基础的问题：

1. 回测里识别出来的结构，是否真的是可复核的 Wyckoff 候选。
2. 模拟盘里同一套候选，是否能被 ptrade 稳定执行、记录和恢复。

只有这两个问题都通过，才有资格继续讨论 L2 / 逐笔强确认、风控闸门和受控自动化执行。
