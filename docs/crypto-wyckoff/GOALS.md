# BTC Wyckoff 2.0 目标与成功准则

## 最终目标

建立一套 BTC Wyckoff 2.0 市场结构系统：用宏观结构识别 Wyckoff 阶段，用微观订单流和衍生品数据验证 Spring / SOS / LPS 的真伪，先支持历史回放和 paper trade，最后才评估受控实盘执行。

最终系统必须能把下面这句话变成可复核的数据对象：

> BTC 跌破关键支撑后，如果清算和仓位重置完成、价格重新收回区间、现货需求没有崩坏，并且 Phase D 的 LPS 回踩供给枯竭，才允许进入右侧交易候选。

## 非目标

- 不做左侧抄底机器人。
- 不以单根插针最低点作为回测买入价。
- 不在第一阶段实现真实交易所下单。
- 不把 P&F 目标预测当成 Spring 真伪判断。
- 不依赖单一交易所的现货价格解释全市场结构。

## 成功准则

### SC-1 统一数据源决策

完成标准：

- 至少验证一个主数据源可以覆盖 BTC 现货 / 永续的逐笔成交、订单簿、OI、Funding 和清算数据。
- 明确这个主数据源是否同时支持历史回放和实时订阅。
- 明确数据延迟、历史深度、费用、权限、限频和本地落盘方案。
- 明确哪些指标需要 fallback 数据源，不能把供应商营销页当成已验证能力。

失败条件：

- 只能拿到 K 线，不能拿到 tick / order book / 衍生品指标。
- 只能看 dashboard，不能以 API 方式稳定落盘。
- 无法对齐毫秒或秒级时间戳。

### SC-2 市场传感器

完成标准：

- 能同时记录 BTC-USDT 现货和永续合约的成交流。
- 能记录至少一档可用订单簿深度，理想状态为 50 档以上。
- 能记录 OI、Funding 和 liquidation events。
- 每条记录包含 `venue`、`instrumentType`、`symbol`、`eventTime`、`receivedAt` 和 `source`。
- 本地采集器有心跳检测、断线重连和延迟指标。

失败条件：

- 数据只在内存里展示，不能回放。
- 网络抖动时静默丢数据。
- 现货和永续数据无法按时间窗口 join。

### SC-3 Phase C 洗盘过滤器

完成标准：

- 当价格跌破结构支撑时，系统输出 `washoutCandidate`，而不是直接输出买入信号。
- 候选对象必须包含结构支撑、跌破幅度、收回速度、清算规模、OI 变化、Funding 状态、现货 CVD、永续 CVD 和盘口恢复。
- 系统能区分 `spring_candidate`、`breakdown_risk`、`short_squeeze_only` 和 `insufficient_evidence`。
- 没有右侧收回和 LPS 复核时，不生成可交易动作。

失败条件：

- 只因价格跌破后反弹就标记 Spring。
- 只用清算峰值判断洗盘完成。
- 不能解释为什么一个候选被拒绝。

### SC-4 历史回放与模拟回测

完成标准：

- 支持 tick 或亚分钟级回放。
- 回测不能在插针最低点假设成交，必须使用收回区间后的可执行价格。
- 支持滑点、手续费、盘口深度和订单延迟假设。
- 报告明确区分结构信号、微观确认、候选拒绝原因和实际可执行点。
- P&F 目标只能在结构通过后参与盈亏比测算，且盈亏比低于 3:1 时归类为无效候选。

失败条件：

- 使用 K 线低点作为买入价。
- 没有滑点模型。
- 无法复现实盘可观测的订单流背景。

### SC-5 Paper Trade 与执行闸门

完成标准：

- 只在 Phase D 的 LPS 右侧回测点允许 paper trade。
- 每笔 paper trade 记录入场理由、失效条件、止损、目标、订单簿环境和数据新鲜度。
- 止损锚定 Phase C 洗盘低点下方，并在 Phase E 后转为追踪止盈。
- 连续样本通过后，才允许评估交易所 sandbox。

失败条件：

- 没有人工审批或风控闸门就接真实账户。
- 延迟、断线或数据缺失时仍继续生成交易动作。
- paper trade 报告无法解释盈亏来自结构判断、执行滑点还是市场噪声。

## 第一阶段北极星指标

第一阶段只看一个核心指标：

> 对历史 BTC Phase C 插针场景，系统能否稳定区分“真实洗盘后收回”与“破位后继续下杀”。

如果这个指标不成立，后续 P&F 目标、仓位管理和自动执行都没有意义。
