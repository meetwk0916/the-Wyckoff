# QMT 接入路线说明

## 背景

结合 QMT 官方知识库、XtQuant 文档，以及新增的客户端系统手册，QMT 这条路线比 PTrade 更接近“本地客户端 + 外部适配器”的工程模式。

这并不意味着 QMT 已经确定优于 PTrade，而是说明它值得被保留为独立实验分支。

截至 2026-05-13，MiniQMT / QMT 路线已经初始化为独立工作面：

- `docs/miniqmt-wyckoff/README.md`
- `docs/miniqmt-wyckoff/ADAPTER-CONTRACT.md`
- `miniqmt-workspace/README.md`

后续实施以 `docs/miniqmt-wyckoff/` 和 `miniqmt-workspace/` 为准；本文保留为前期接入路线研究记录。

## 本次新增参考资料

- `docs/reference/qmt/迅投QMT极速策略交易系统说明文档.pdf`

这份手册补充了之前网页文档没有完全展开的几个事实：

1. QMT 客户端内置模型研究和模型交易能力，支持 `VBA` 与 `Python`。
2. 模型交易支持“模拟”与“实盘”切换，实盘模式会向柜台发送真实交易信号。
3. 客户端界面直接展示用户 `Level2` 权益。
4. Level2 界面包含委托队列、逐笔委托、逐笔成交、千档盘口和十档盘口等能力。
5. 客户端允许外部系统通过 `DBF` / `CSV` / 共享文件方式与交易端交互。
6. 管理端和交易端存在白名单与风控约束，包括账号白名单、增强交易白名单和 Python 库白名单。

## 现阶段对 QMT 的判断

### 结论一：QMT 至少有两层能力面

第一层是客户端内能力：

- 模型研究
- 回测
- 模型交易
- 模拟 / 实盘切换
- Level2 图形与盘口观察

第二层是外部接入能力：

- XtQuant 外部 Python 库
- 共享文件交互方式
- 数据导出与主推回报文件更新

### 结论二：对当前仓库更有价值的是第二层

当前仓库已经有自己的前端和 Node bridge，因此并不需要把 QMT 客户端本身搬进来。

更有价值的是把 QMT 当成真实行情 / 交易运行时，再在 Windows 侧做一个适配器，把标准化后的结果送给当前仓库。

## 推荐实现方式

### 优先方案：XtQuant 外部 Python 适配器

核心形态：

`MiniQMT / QMT -> XtData + XtTrader -> Windows 本地适配器 -> relay -> 当前 Node bridge`

原因：

- XtData 明确支持行情订阅、全推、历史数据和 Level2 数据类型。
- XtTrader 明确支持连接、订阅、报单、撤单、资产 / 持仓 / 成交查询和主推回调。
- 官方完整示例本身就是独立 Python 脚本连接 `userdata_mini` 或 `userdata`。
- 这种模式最接近标准 SDK 适配器，更容易做日志、重连、录制、健康检查和标准化封装。

推荐最小实现：

1. Windows 常驻进程启动 XtData / XtTrader。
2. 连接 MiniQMT 或 QMT 客户端。
3. 验证账户状态、行情连接和 Level2 权限。
4. 输出统一健康检查 JSON。
5. 输出统一 L2 order-flow JSON。
6. 后续再逐步增加成交、委托、持仓和执行反馈事件。

### 降级方案：共享文件交互

核心形态：

`外部系统写入 DBF/CSV -> QMT 客户端监控并下单 -> 委托/成交/持仓回报写回共享文件 -> relay / bridge 读取`

这是本次 PDF 提供的新线索。

从系统手册看，QMT 客户端支持：

- 批量埋单
- 批量预埋单
- CSV 预埋单
- 共享文件方式下发委托与读取回报

这条路线更像“文件驱动交易网关”，适合作为降级执行通道，但不适合作为主行情通道。

原因：

- 文件交互对实时行情和 L2 标准化不够自然。
- 适合委托下发与回报消费，不适合做高质量行情适配。
- 运维和一致性处理会比 XtQuant 适配器更重。

因此它更适合作为执行 fallback，而不是主接入模式。

## 与 PTrade 路线的差异

### PTrade 更像运行时内导出

- 逻辑运行在券商给定策略运行时中。
- 更适合在运行时内直接取快照、逐笔和账户信息。
- 对运行时环境权限和网络策略更敏感。

### QMT 更像本地客户端适配

- 逻辑运行在 Windows 本地 Python 进程中。
- 更适合独立构建行情 / 交易 / 回调适配器。
- 更方便做本地服务化、重连管理和中间契约层。

## 当前建议

### 建议一：QMT 路线先按外部适配器设计，不按客户端内脚本设计

客户端内脚本当然可用，但对当前仓库来说，真正需要的是一个可复用、可桥接、可录制的外部适配层。

### 建议二：优先把 QMT 当作 Windows 侧能力，不要尝试在当前 WSL 中直接模拟

QMT / MiniQMT 的真实连接前提在 Windows 侧，当前 WSL 仓库不适合直接承担客户端运行时。

### 建议三：共享文件交互保留为备用方案

如果未来实际券商环境对 XtQuant 外部 Python 有限制，或者某些交易能力必须经过 QMT 客户端共享文件链路，则可以用 DBF / CSV / 导出回报作为降级执行方案。

## 最小验证建议

### QMT 验证步骤 1：环境可达性

- 确认 Windows 侧已安装 MiniQMT 或 QMT。
- 确认账号可登录。
- 确认目标账户具有所需权限。

### QMT 验证步骤 2：XtQuant 基线能力

- 建立 `XtQuantTrader(path, session_id)` 连接。
- 查询 `query_account_infos()` / `query_account_status()`。
- 订阅账户回调。
- 拉取 `get_full_tick()` 或 `subscribe_quote()`。

### QMT 验证步骤 3：Level2 能力

- 检查目标账号的 `Level2` 权益。
- 验证 `l2quote`、`l2order`、`l2transaction` 等数据是否可读。
- 记录真实字段、时间戳和订阅稳定性。

### QMT 验证步骤 4：桥接输出

- 把健康状态输出为统一 JSON。
- 把 order-flow 输出为统一 JSON。
- 通过 HTTP 或文件形式送入当前 Node bridge。

## 当前结论

1. QMT 已初始化为独立 MiniQMT 工作面，不再只作为一句“以后可能支持”的占位描述。
2. 对当前仓库最可实现的 QMT 路线是 `XtQuant 外部 Python 适配器`。
3. 新增 PDF 说明共享文件交互也是官方支持路径，可作为执行降级方案。
4. 在没有 Windows 侧真实环境验证前，QMT 仍然处于“可实现但未验证”的状态。
