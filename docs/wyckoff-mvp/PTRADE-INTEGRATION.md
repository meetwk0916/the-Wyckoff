# ptrade 对接分阶段说明

## 背景

当前产品路线已经调整：ptrade 不是一个最后才补的边缘适配器，而是 Wyckoff 2.0 后续能力里的关键基础设施。

但当前优先级已经进一步收口：先在 ptrade 官方运行时内部打通“回测 -> 模拟盘 -> 交易报告”的最小闭环，再把 L2 订单流作为信号增强能力补回同一条链路。

这意味着此前的 L2 验证仍然重要，但它现在更像 Phase 0 的环境预检查，而不是当前阶段的最终交付物。

## 当前仓库状态

- 已落地本地 `ptrade bridge` 服务骨架
- 已提供 L2 订单流样例接口和前端联调面板
- 已验证 ptrade 运行时账号绑定、本地落盘和基础网络边界
- 已补充最小环境验证脚本 `ptrade_phase1_validation.py`
- 已新增 ptrade 原生策略骨架 `ptrade_wyckoff_trader.py`
- 已新增回测 / 模拟盘操作说明 `PTRADE-TRADING.md`
- 当前前端默认仍是 `mock` 模式，不代表真实 ptrade 已接通

## 官方文档确认后的推荐路径

结合官方帮助中心、策略示例、现货接口和业务公共接口，当前更推荐的实现方式是：

1. 先用 `ptrade_phase1_validation.py` 做 Phase 0 环境预检查。
2. 再用 `ptrade_wyckoff_trader.py` 直接在 ptrade 内跑回测和模拟盘。
3. 策略内优先使用官方已经明确支持回测 / 交易的接口：
	- `order`
	- `cancel_order`
	- `get_open_orders`
	- `get_orders`
	- `get_trades`
	- `get_position`
	- `get_positions`
4. 每个交易日结束把策略状态、订单、成交和持仓汇总为 JSON 报告，落到 `/home/fly/notebook/`。
5. 等回测和模拟盘闭环稳定后，再把 L2 订单流重新接回到信号确认链路。

这样做的原因是：

- 官方客户端本身已经把“新建回测”和“新建交易”作为一等能力。
- 现货接口里的 `order(...)` 明确支持回测和交易。
- `order_target` / `order_target_value` 虽然也支持回测和交易，但官方文档明确提示交易场景可能因为持仓同步滞后导致重复下单，因此当前骨架优先使用 `order` 做 delta 下单。
- 交易详情界面天然就有评价指标、交易明细、持仓明细和交易日志，适合先把最小闭环跑通。
- L2 仍然重要，但它应该增强策略质量，而不是阻塞回测 / 模拟盘基础能力落地。

当前仓库中与这条路径直接相关的文件有：

- `ptrade_phase1_validation.py`
- `PTRADE-VALIDATION.md`
- `ptrade_wyckoff_trader.py`
- `PTRADE-TRADING.md`

## Phase 0：环境预检查

当前定位：已具备最小可用脚本，应当继续作为上线前检查。

### 目标

- 验证账号绑定是否正确。
- 验证本地落盘是否可用。
- 验证 L2 权限和网络边界是否满足下一步联调。

### 关键交付物

- `ptrade_phase1_validation.py`
- `PTRADE-VALIDATION.md`

### 完成标准

- 能拿到登录账号和绑定账号。
- 能把结果写到 `/home/fly/notebook/`。
- 能区分 L2 未授权、非交易时段和网络不可达这几种情况。

## Phase 1：ptrade 原生回测 + 模拟盘 + 交易报告

当前定位：首要优先级。

### 目标

- 在 ptrade 内使用同一份策略骨架同时支撑回测和模拟盘。
- 形成可复核的日终交易报告，用于后续实盘跟踪。
- 在不引入自动实盘的前提下，把信号、仓位、订单和成交放进同一条闭环里。

### 关键交付物

- `ptrade_wyckoff_trader.py`
- `PTRADE-TRADING.md`
- 基于 `order` / `get_open_orders` / `get_trades` / `get_positions` 的最小执行闭环
- 落盘到 `/home/fly/notebook/ptrade-wyckoff-trade-report-last.json` 的交易报告

### 明确不做

- 无人工闸门的实盘执行
- 多券商统一执行抽象层
- 完整绩效平台
- 最终版 Wyckoff 信号模型

### 完成标准

- 同一份策略代码可以在回测中运行。
- 同一份策略代码可以在模拟盘中运行并提交订单。
- 日终可以拿到包含信号、决策、订单、成交和持仓的 JSON 报告。
- 通过 `get_open_orders` 避免同一标的重复叠单。

## Phase 2：L2 订单流增强与统一契约

当前定位：在 Phase 1 稳定后推进。

### 目标

- 把 L2 订单流从“可验证”提升为“可持续增强信号质量”的输入。
- 让 ptrade 内信号判断和外部 bridge / 前端面板消费同一类标准化结果。

### 关键交付物

- L2 订单流统一数据契约
- 订单流录制与回放能力
- 策略内可选的 L2 确认逻辑
- bridge 真实上游连接与状态展示

### 完成标准

- L2 数据不再只是单次验证脚本输出。
- L2 可以被策略侧和前端侧一致消费。
- 可以清楚地区分 mock、validation、live ptrade 三类状态。

## Phase 3：实盘执行与风控闸门

当前定位：最后阶段。

### 目标

- 在回测、模拟盘和 L2 增强都稳定后，再进入实盘执行。
- 把审批、风控、失败恢复和审计放在下单之前，而不是之后补救。

### 关键交付物

- 实盘模式显式开关
- 下单、撤单、查单与执行反馈闭环
- 人工审批与风控闸门
- 可审计的执行日志与失败恢复策略

### 完成标准

- 策略不会因为“只是把模拟盘切成实盘”就直接进入危险状态。
- 风控和审批是显式配置，而不是隐含约定。
- 实盘路径建立在回测 / 模拟盘 / L2 都已验证过的基础之上。

## 当前结论

1. `ptrade_phase1_validation.py` 现在应视为 Phase 0 预检查脚本，而不是最终策略方案。
2. 当前最优先要完成的是 ptrade 原生的“回测 + 模拟盘 + 交易报告”闭环。
3. L2 订单流仍然重要，但它应该作为 Phase 2 的信号增强输入，而不是继续阻塞 Phase 1。
4. 实盘执行必须继续放在最后，并且要带着明确的风控闸门进入。
