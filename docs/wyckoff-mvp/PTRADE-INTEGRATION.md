# ptrade 对接分阶段说明

## 背景

当前产品路线已经调整：ptrade 不是一个最后才补的边缘适配器，而是 Wyckoff 2.0 后续能力里的关键基础设施。

按照当前判断，最先需要拿到的不是自动下单，而是 L2 订单流数据。原因是这类数据更直接影响 spring、BUEC、LPS 回踩、承接质量和盘口吸收等判断是否能从结构分析推进到盘口验证。

因此，ptrade 对接按三个阶段推进，顺序不能倒置。

## Phase 1：L2 订单流接入

当前定位：首要优先级。

### 目标

- 从 ptrade 环境稳定获取 L2 订单流相关数据。
- 把原始返回整理为 Wyckoff 2.0 可消费的统一数据契约。
- 为后续研究、验证和回放建立稳定输入面。

### 关键交付物

- ptrade 连接、权限、标的订阅与健康检查能力
- L2 订单流适配层
- 统一的订单流数据契约
- 本地录制与回放能力
- L2 数据的新鲜度、延迟与可用性状态

### 明确不做

- 自动交易
- 实盘下单
- 回测系统全量能力
- 执行反馈闭环

### 完成标准

- 目标标的可以稳定采集 L2 订单流数据
- 数据可以被统一结构消费，而不是直接耦合 ptrade 原始返回
- 本地可以对采集结果做回放，用于 Wyckoff 结构与盘口验证

## Phase 2：其他 API 能力与回测

当前定位：在 Phase 1 稳定后推进。

### 目标

- 扩展除 L2 订单流外的其他 API 能力。
- 为回测、复盘和研究接口建立可持续的输入边界。

### 关键交付物

- 历史数据、回放数据或等价研究接口
- 查询类接口梳理，例如账户、持仓、委托、成交等能力边界
- 面向回测与研究的数据导出或重放链路
- 研究接口、查询接口与执行接口的分层说明

### 完成标准

- 回测和研究链路能够消费前一阶段沉淀的数据能力
- 其他 API 需求与自动交易逻辑明确分层，不相互污染
- 文档中可以清晰说明哪些是研究能力，哪些是执行能力

## Phase 3：自动交易与执行闭环

当前定位：最后阶段。

### 目标

- 在 Phase 1 和 Phase 2 稳定后，再推进半自动或自动交易能力。

### 关键交付物

- 下单、撤单、查单与执行反馈闭环
- 人工审批与风控闸门
- 模拟模式到实盘模式的切换策略
- 执行日志、失败恢复与审计能力

### 完成标准

- 可以在同一工作流中看到信号、审批、执行与反馈
- 风控不是隐含逻辑，而是明确的系统闸门
- 自动交易建立在前两阶段稳定的前提上，而不是跳过研究与验证直接接单

## 当前结论

1. ptrade Phase 1 的 L2 订单流接入，应该进入首要完成的功能清单。
2. 其他 API 能力与回测，应当排在 L2 订单流稳定之后。
3. 自动交易必须放在最后，不能倒置为当前阶段核心目标。
