# 券商接入实验分支总览

## 目的

当前仓库已经完成了前端控制台、本地 ptrade bridge 骨架和最小 ptrade 校验脚本。

下一阶段需要把真实券商接入收敛成两条可验证、可比较、可回退的实验分支，而不是在一个实现里同时混入多种运行时假设。

这里的“分支”指技术实验路线，不是 Git 分支名。

## 当前结论

当前保留两条实验分支，等待后续可行性验证结果再决定主路线：

### 实验分支 A：PTrade 运行时导出路线

核心形态：

`PTrade 交易运行时策略 -> HTTP relay 或本地落盘 -> 当前 Node bridge -> 前端`

当前判断：

- 更适合已有 ptrade 账号、已在券商运行环境内完成登录和权限开通的场景。
- 更像“运行时内策略导出”，而不是外部程序主动拉取。
- 对 L2、账号绑定、运行时网络策略的依赖更强。

已完成材料：

- `PTRADE-INTEGRATION.md`
- `PTRADE-VALIDATION.md`
- `ptrade_phase1_validation.py`

当前验证重点：

1. 账号绑定是否可见。
2. Level2 是否真实可读。
3. 出站 HTTP 是否允许。

### 实验分支 B：QMT Windows 本地适配器路线

核心形态：

`Windows MiniQMT / QMT + XtQuant 适配器 -> HTTP relay 或本地落盘 -> 当前 Node bridge -> 前端`

当前判断：

- 工程实现上更接近标准本地 SDK 适配器模式。
- 更适合把行情、交易、回调统一封装到一个 Windows 常驻进程里。
- 与当前仓库的 bridge 结构更自然衔接。

已完成材料：

- `QMT-INTEGRATION.md`
- `docs/reference/qmt/迅投QMT极速策略交易系统说明文档.pdf`

当前验证重点：

1. 实际券商环境能否稳定运行 MiniQMT / QMT。
2. 是否可用 XtQuant 外部 Python 方式接入。
3. Level2 与逐笔能力在目标账号上是否真实可用。

## 两条路线的统一约束

无论最终选哪条路线，当前仓库都保持同一个外部契约：

1. 券商运行时或客户端侧负责真实数据采集。
2. 中间适配层负责标准化、健康检查、录制和回放。
3. 当前仓库中的 Node bridge 继续承担统一 HTTP 契约层。
4. 前端只消费统一数据契约，不直接耦合券商原始返回。

当前最小契约仍然是：

- `GET /api/ptrade/health`
- `GET /api/ptrade/l2-order-flow?symbol=...`

后续如果接入 QMT，接口名可以暂时保留不变，先保证前端与中间层不需要大改；等底层稳定后，再统一收敛命名。

## 为什么现在不直接二选一

当前证据还不完整：

- PTrade 已有账号和明确的运行时路线，但真实网络策略与 L2 可用性仍待现场验证。
- QMT 官方资料显示外部适配更直接，但是否能在你的实际券商环境里稳定落地，仍需结合真实账号和客户端版本验证。

因此，当前更合理的做法不是提前锁死，而是保留两条实验分支，按最小验证顺序推进。

## 推荐验证顺序

### 第一优先级：先完成 PTrade 最小验证

原因：

- 你已经明确有 ptrade 账号。
- 仓库内最小验证脚本已经准备好。
- 这是最贴近真实可运行环境的一条路线。

验收门槛：

- 能看到绑定账号。
- 能确认到 Level2 线索。
- 能确认 HTTP 可直连或必须改为落盘中转。

### 第二优先级：并行准备 QMT Windows 侧验证

原因：

- QMT 更像长期可工程化的本地适配器路线。
- 官方文档显示其外部 Python 能力、行情订阅、交易回调和共享文件交互都较成熟。

验收门槛：

- Windows 侧能稳定连接 MiniQMT / QMT。
- 可以拿到账户状态、行情订阅、逐笔或 Level2 数据。
- 可以把标准化结果向外转发到 relay 或文件。

## 落地建议

### 如果 PTrade 先通过验证

优先推进：

`PTrade exporter -> relay -> bridge -> frontend`

同时保留 QMT 为备选执行适配层，不立即废弃。

### 如果 PTrade 卡在运行时网络或权限

优先推进 QMT：

`QMT XtQuant adapter -> relay -> bridge -> frontend`

PTrade 保留为补充接入，不再作为唯一主路线。

### 如果两边都能跑通

优先统一：

- 数据契约
- 健康检查模型
- 录制 / 回放格式
- 执行与查询事件模型

然后再按可维护性、延迟、权限成本和实盘稳定性决定默认接入路线。

## 当前仓库建议

1. 保持前端只依赖统一 bridge 接口。
2. PTrade 继续走运行时内导出和最小验证。
3. QMT 优先按 XtQuant 外部 Python 适配器路线设计。
4. 如果 QMT 目标环境对外部 Python 受限，再考虑共享文件交互作为降级方案。
